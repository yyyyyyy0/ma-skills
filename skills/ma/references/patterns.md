# ma High-risk Default Patterns

これらのパターンは、既定では避けるべき高リスク表現である。
醜いから禁止するのではなく、原則違反になりやすく、説明責任なしに採用すると
情報構造や操作理解を崩しやすいからである。
使うなら「何を伝えているか」を明示し、例外採用の理由を出力内に残すこと。

- **Glassmorphism**
  - 失敗しやすい理由: ぼかしが情報の境界とグループの帰属を曖昧にしやすい
  - 許容条件: 背景との差分、重なり順、固定レイヤーの関係を明確に示せるとき
  - 説明義務: どのレイヤー関係または状態境界を伝えているかを書く
- **アイコンとしての絵文字**
  - 失敗しやすい理由: 文脈と文化で意味が揺れ、操作対象の安定した記号になりにくい
  - 許容条件: 既存ブランドやコミュニティ文脈で意味が固定され、補助ラベルが付くとき
  - 説明義務: 絵文字単体に頼らず何の意味を補助しているかを書く
- **主テーマとしてのグラデーション**
  - 失敗しやすい理由: 雰囲気だけを増やし、情報階層やカテゴリを伝えないまま面積を支配しやすい
  - 許容条件: ブランド制約、密度表現、カテゴリ差、データの連続量を伝える役割があるとき
  - 説明義務: どの連続性や区別を伝えるために使うかを書く
- **装飾的な影**
  - 失敗しやすい理由: 階層や高さの意味を持たない奥行きが、重要度と物理感を誤読させやすい
  - 許容条件: 固定ヘッダー、ドラッグ対象、浮遊レイヤー、優先面の前後関係を示すとき
  - 説明義務: どの高さ関係または操作可能性を示しているかを書く
- **純粋に装飾的なホバーエフェクト**
  - 失敗しやすい理由: アフォーダンスや状態変化を伴わない動きが、意味のない刺激になる
  - 許容条件: クリック可能性、選択状態、因果関係、プレビュー対象を明確にする場合
  - 説明義務: ホバーで何の状態差分を伝えるかを書く
- **雰囲気としての色**
  - 失敗しやすい理由: カテゴリ、状態、優先度、関係性を持たない色が認知ルールを濁らせる
  - 許容条件: ブランド制約があり、同時に状態・カテゴリ・強調の規則としても一貫運用できるとき
  - 説明義務: その色が何を区別し、どの規則で再利用されるかを書く
- **構造的理由のない丸いカード**
  - 失敗しやすい理由: 柔らかさだけを足して関係や境界の意味を説明しない
  - 許容条件: タップ領域、グループ境界、状態コンテナとして角の意味が明確なとき
  - 説明義務: なぜその角丸が必要で、何の境界を伝えるかを書く
- **opacity-based text color**
  - 失敗しやすい理由: 背景との関係が不安定で、コントラスト設計を壊しやすい
  - 許容条件: 実背景込みで測定し、全状態で十分なコントラストが担保できるとき
  - 説明義務: どの背景で検証し、どの帯域に置くかを書く
- **flat contrast scale**
  - 失敗しやすい理由: どのテキストも似た濃さになり、優先度が潰れる
  - 許容条件: 意図的にフラットな読ませ方をする限定的な場面で、開始点が別手段で明確なとき
  - 説明義務: どの要素が主従を担保するかを書く
- **contrast as decoration**
  - 失敗しやすい理由: 「柔らかく見せたい」だけの低コントラストは意味を伝えない
  - 許容条件: 情報優先度や状態の差を、他の明確な信号と組み合わせて説明できるとき
  - 説明義務: 低コントラスト化で何を伝え、何で補完するかを書く
- **uniform spacing throughout**
  - 失敗しやすい理由: どこも同じ余白だと、どこがまとまりか分からない
  - 許容条件: 同じ意味レベルの反復構造だけで構成される場面
  - 説明義務: どのまとまりが等価で、どこで階層を立てるかを書く
- **arbitrary spacing values**
  - 失敗しやすい理由: その場しのぎの数値は偶発的な階層を生む
  - 許容条件: 既存 scale では説明できない例外があり、再利用基準を明文化するとき
  - 説明義務: なぜ既存 scale では足りず、その値が何を示すかを書く
- **near-identical spacing variants**
  - 失敗しやすい理由: 意味の差がない微差はノイズになる
  - 許容条件: 反復密度を段階制御する明確なルールがあるとき
  - 説明義務: どの差分がどの意味段階に対応するかを書く
- **spacing as decoration**
  - 失敗しやすい理由: 「きれいに見えるから」だけで gap を足すと関係の読みが崩れる
  - 許容条件: 余白がグルーピング、焦点化、密度制御のいずれかを担うとき
  - 説明義務: その gap が何の関係を伝えるかを書く
- **empty space without function**
  - 失敗しやすい理由: 分離・焦点化・密度調整を説明できない空白は意味を持たない
  - 許容条件: 視線の休止、重要ブロックの隔離、スクロールの節目として機能するとき
  - 説明義務: その空白がどの読み順または認知負荷制御に効くかを書く
- **collapsed hierarchy**
  - 失敗しやすい理由: 異なる意味レベルに同じ余白や同じコントラストを使い、構造を潰す
  - 許容条件: フラットであること自体が機能要件で、別の信号で優先順位を保証できるとき
  - 説明義務: どの別信号で階層を補うかを書く
- **Proxy worship**
  - 失敗しやすい理由: metric が設計理由そのものとして扱われ、意味設計が空洞化する
  - 許容条件: semantic role と構造が先に定義され、metric はその作用確認にだけ使われるとき
  - 説明義務: 何を原理で決め、何だけを観測したのかを書く
- **Pixel-level empiricism**
  - 失敗しやすい理由: token や styling の微差を、意味仮説なしにテストして局所最適化へ落ちる
  - 許容条件: 同一の意味を持つ整合的な案同士で、経験的な不確実性だけが残っているとき
  - 説明義務: 比較対象が同じ semantic role を満たしていることを書く
- **Local metric optimization**
  - 失敗しやすい理由: 局所指標の改善を優先して system coherence、reading order、mapping clarity を壊す
  - 許容条件: 局所改善が全体構造を壊さず、他軸の原則違反を生まないと確認できるとき
  - 説明義務: 何を改善し、どの原則を壊していないかを書く
- **Measurement before meaning**
  - 失敗しやすい理由: role、target、hierarchy を定義する前に比較を始めるため、問いの種類が崩れる
  - 許容条件: 役割、対象、優先順位が文章で明示され、その後の不確実性だけを測るとき
  - 説明義務: 何が conceptual question で、何が empirical question なのかを書く
