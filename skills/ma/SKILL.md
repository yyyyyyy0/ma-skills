UIデザインの根拠となる2つの原則は、情報の明瞭性と目的ある装飾である。
すべての視覚的要素は、自分が何を伝えているかで存在を正当化しなければならない。

## When to Use

- 新しいUIを設計・実装したいとき
- 既存UIを、情報設計から組み直したいとき
- 画面構造、状態、優先順位、CTA の順序を整理したいとき
- ただ美しくするのではなく、意味が伝わる見た目にしたいとき
- 実装方針とデザイン根拠を同時に出したいとき

## Do Not Use

- 完成UIの総合監査だけが目的のとき。`ma-review` を使う
- 可読性やアクセシビリティだけを点検したいとき。`ma-legibility` を使う
- 「この Delete は何を消すのか」など、作用対象の明確化だけが目的のとき。`ma-mapping` を使う
- 既存スタイルの不整合や token drift だけを点検したいとき。`ma-system` を使う

## Design Philosophy

1. **UIは人間の認識と操作のために最適化されなければならない。**
   情報の階層・レイアウト・インタラクションパターンは、人が自然に知覚し行動する方法から
   生まれるべきであり、慣習やトレンドからではない。
   直感性が目標である。ユーザーは行動する前に考えるべきではない。

2. **装飾は情報のために存在する。**
   色、余白、影、モーション、タイポグラフィ、サイズは
   「これは何を伝えているか？」に答えなければならない。
   答えられなければ、その要素は存在すべきではない。
   「見た目がいい」は正当な理由ではない。「Xを伝えている」が正当な理由だ。

## Workflow

1. エンティティ・状態・関係を整理する
2. ユーザーが認識すべき認知順序を定義する
3. その順序を支える最小の視覚システムを決める
4. 実装方針を出す
5. 完成物レビューが必要なら `ma-review` に渡す

### 実施手順

- まず、画面が何を扱うかを定義する
  - エンティティは何か
  - どんな状態があるか
  - エンティティ間の関係は何か
- 次に、ユーザーが何をどの順で理解すべきかを定義する
- その後で、色・余白・階層・動き・文字サイズを決める
- 最後に、実装時に必要なコンポーネント構成、状態分岐、主要アクションを明文化する

## Constraints

- 視覚的階層は情報の優先度を直接反映しなければならない
- 次に可能なアクションは説明なしに自明でなければならない
- レイアウトは認知的フローに従わなければならない
- 色の使用には、状態・カテゴリ・優先度・関係性・フィードバックのいずれかの目的が必要
- アニメーションとモーションは状態遷移または因果関係を伝えなければならない
- タイポグラフィ階層はグレースケールでも読めなければならない
- 余白はグルーピングを伝えるために使うこと
- 完成物の監査はこのスキルで抱え込まず、`ma-review` へ渡すこと

## Negative List

これらのパターンは醜いから拒否されるのではなく、哲学に失敗するから拒否される。

- **Glassmorphism**: ぼかしは情報の境界とグループの帰属を曖昧にする
- **アイコンとしての絵文字**: 文脈と文化によって意味が不安定
- **主テーマとしてのグラデーション**: コンテンツについて何も伝えない
- **装飾的な影**: 階層や高さの意味を持たない奥行き
- **構造的理由のない丸角カード**: 何も伝えない柔らかさ
- **純粋に装飾的なホバーエフェクト**: アフォーダンスや状態変化を伝えない
- **雰囲気としての色**: カテゴリ的または状態的な意味を持たない色

## Sub-skills

`ma` は設計・実装のスキルである。審査・改善は専用サブスキルに委ねる。

| スキル | 用途 |
|---|---|
| `ma-review` | 完成UIの総合監査 |
| `ma-system` | デザインシステムの一貫性 |
| `ma-legibility` | 可読性・アクセシビリティ |
| `ma-mapping` | コントロールとターゲットの関係 |
| `ma-flow` | 視線の流れと情報順序 |
| `ma-reduction` | 不要な複雑さの除去 |

## Output Contract

このスキルの出力は、次のどちらかのモードで返すこと。

### 1. 実装指示モード

- 画面またはコンポーネントの目的
- 情報構造
- 主要状態
- CTA とその優先順位
- 実装上の構成案
- 見た目の判断理由

### 2. デザイン根拠サマリモード

- 何を先に見せるか
- 何を後に見せるか
- なぜその視覚ルールが必要か
- 何を削るべきか
- どのサブスキルで次にレビューすべきか

## Examples

**Positive example 1**

入力: 「分析ダッシュボードを設計して。数値だけでなく、何を見るべきかが分かるようにしたい」

期待: エンティティ、状態、優先順位、主要CTAを整理したうえで、実装方針まで返す。

**Positive example 2**

入力: 「既存の設定画面がごちゃついてる。情報設計から組み直したい」

期待: 設定カテゴリの再編、認知順序、段階的開示、削るべき要素を含めた設計指示を返す。

**Positive example 3**

入力: 「ワイヤーから React 実装の方向性までまとめて」

期待: 画面構造とコンポーネント責務を整理し、視覚判断の理由も添える。

**Negative example**

入力: 「このUIをレビューして、問題点を high / mid / low で出して」

対応: このスキルでは抱えず、`ma-review` を使う。

## Growing This Skill

生成された出力が哲学に反する新しいパターンを示したとき:
1. どの原則が違反されたかを特定する
2. なぜ失敗するかを述べる
3. Negative List に1行追加する

哲学が審判。Negative List は判例集。
このエージェントは卓越したクリエイティブワークができる。この哲学を完全な信念をもって適用せよ。
