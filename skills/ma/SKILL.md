UIデザインの根拠となる2つの原則は、情報の明瞭性と目的ある装飾である。
すべての視覚的要素は、自分が何を伝えているかで存在を正当化しなければならない。

## When to Use

- 新しいUIを設計・実装したいとき
- 既存UIを、情報設計から組み直して実装方針まで出したいとき
- 画面構造、状態、優先順位、CTA の順序を整理したいとき
- ただ美しくするのではなく、意味が伝わる見た目にしたいとき
- 実装方針とデザイン根拠を同時に出したいとき
- 余白やコントラストを含めて、なぜその見た目なのかまで説明したいとき

## Do Not Use

- 完成UIの総合監査だけが目的のとき。`ma-review` を使う
- 可読性やアクセシビリティだけを点検したいとき。`ma-legibility` を使う
- 「この Delete は何を消すのか」など、作用対象の明確化だけが目的のとき。`ma-mapping` を使う
- 既存スタイルの不整合や token drift だけを点検したいとき。`ma-system` を使う
- 単純に見た目だけを派手にしたいが、情報設計は触らないと決まっているとき

## Design Philosophy

1. **UIは人間の認識と操作のために最適化されなければならない。**
   情報の階層・レイアウト・インタラクションパターンは、人が自然に知覚し行動する方法から
   生まれるべきであり、慣習やトレンドからではない。
   直感性が目標である。ユーザーは行動する前に考えるべきではない。

2. **装飾は情報のために存在する。**
   色、余白、影、モーション、タイポグラフィ、サイズは
   「これは何を伝えているか？」に答えなければならない。
   答えられなければ、その要素は存在すべきではない。
   「見た目がいい」は正当な理由ではない。「Xを伝えている」が正当な理由だ。

3. **余白とコントラストは、雰囲気ではなく構造である。**
   余白はグルーピング、リズム、密度、焦点を伝える。
   コントラストは可読性だけでなく情報優先度を伝える。
   どちらも「整って見えるから」ではなく、「関係が分かるから」採用する。

## Workflow

1. エンティティ・状態・関係を整理する
2. ユーザーが認識すべき認知順序を定義する
3. その順序を支える最小の視覚システムを決める
4. コントラストと余白のルールを先に定義する
5. 実装方針を出す
6. 完成物レビューが必要なら `ma-review` に渡す

## Constraints

- 視覚的階層は情報の優先度を直接反映しなければならない
- 次に可能なアクションは説明なしに自明でなければならない
- レイアウトは認知的フローに従わなければならない
- 色の使用には、状態・カテゴリ・優先度・関係性・フィードバックのいずれかの目的が必要
- アニメーションとモーションは状態遷移または因果関係を伝えなければならない
- タイポグラフィの階層はグレースケールでも読めなければならない
- コントラストは閾値を満たすだけでなく、優先度の差を伝えなければならない
- 同じ意味レベルの要素は、同じコントラスト帯に置くこと
- 余白は装飾ではなくグルーピングとリズムを伝えるために使うこと
- spacing scale をレイアウト前に定義すること
- グループ内の余白はグループ間の余白より狭くすること
- 余白の関係は画面全体で意味的に一貫していなければならない
- 余白のリズムを崩すのは、階層、境界、焦点を伝えるときだけにすること
- negative space は密度制御と重要要素の隔離のために意図的に使うこと
- 2つの領域が同じ余白を持つなら、同じ種類の関係を示していなければならない
- 完成物の監査はこのスキルで抱え込まず、`ma-review` へ渡すこと
- 単一論点の見直しは親スキルで抱え込まず、必要に応じて `ma-*` サブスキルへ寄せること

## Contrast

コントラストは polish ではなく structure である。

### 役割

- **Legibility**: すべての状態で文字が背景に対して読めること
- **Hierarchy**: どこから読ませるか、何が主で何が従かを示すこと

### Rules

- 色を決める前に、意味ごとのコントラスト帯を決めること
- 典型的な帯域の目安:
  - Primary text: 12:1+（見出し、主要数値、主要 CTA ラベル）
  - Body text: 7:1-10:1（本文、説明文）
  - Secondary text: 4.5:1-6:1（ラベル、補足、メタデータ）
  - Disabled / decorative: large text で 3:1 を下回らない。読ませる文字には使わない
- 同じ意味レベルの要素は同じコントラスト帯に置くこと
- 実際の背景に対して測ること。gradient、translucent layer、blur は見かけの背景色を変える
- opacity で文字色を作らないこと。`rgba()` の薄め方は知覚上の明度を安定して制御できない
- 状態変化をコントラスト差だけに頼らないこと。構造、ラベル、タイポグラフィの変化も併用すること

### What Good Looks Like

- 見出しが最初に読まれ、本文が次に続き、補足情報は後景に下がる
- WCAG を満たしていても、すべてが同じ見え方なら失敗とみなす

## Spacing

余白は decoration ではなく structure である。

### 役割

- **Grouping**: 近さで関係を示し、距離で分離を示す
- **Rhythm**: 繰り返される間隔で読み進めるペースをつくる
- **Silence**: 密度を下げ、重要な要素を孤立させて焦点化する

### Rules

- レイアウト前に spacing scale を決めること。恣意的な数値は恣意的な関係を生む
- 4px または 8px grid は目安であり、目的は一貫性である
- 等しい余白は等しい関係を意味する。意味が違う階層に同じ余白を使わない
- ラベルと値のような1ユニット内の余白は、そのユニットと次のグループの間より必ず狭くすること
- intra-group spacing を inter-group spacing より小さく保つこと
- 画面全体で spacing relationship を意味的に一貫させること
- リズムを破るなら、その破れ自体が境界、焦点、セクション転換、状態変化を伝えていなければならない
- 大きな空白は hierarchy、pacing、focus、density reduction のいずれかを説明できなければならない
- negative space は内容がない部分ではなく、内容の読み方を制御する要素として扱うこと

### Negative List

- **Uniform spacing throughout**: equal spacing everywhere implies equal relationship. If everything is equidistant, nothing is grouped.
- **Arbitrary spacing values**: ad hoc numbers create accidental hierarchy and unstable structure.
- **Near-identical spacing variants**: tiny differences without semantic distinction create noise, not nuance.
- **Spacing as decoration**: gaps added only to make the layout feel "clean."
- **Empty space without function**: blank areas that do not isolate, pace, separate, or reduce density.
- **Collapsed hierarchy**: identical spacing across different semantic levels.

### Spacing Example

Bad:
- Title, subtitle, body, CTA, and next section all use the same spacing.

Why it fails:
- Equal spacing implies equal relationship.
- The layout loses grouping, rhythm, and focus.

Better:
- Keep title and subtitle close.
- Separate the content block from the CTA more clearly.
- Add larger space before the next section.
- Use isolation only where emphasis is intentional.

### What Good Looks Like

- タイトルとサブタイトルは近く、セクション間は明確に離れている
- CTA は本文の続きであることが分かる距離か、独立した意思決定点であることが分かる距離になっている
- 情報量が多い場面でも、余白の階層で読む順序が保たれている

## High-risk Default Patterns

これらのパターンは、既定では避けるべき高リスク表現である。
醜いから禁止するのではなく、原則違反になりやすく、説明責任なしに採用すると
情報構造や操作理解を崩しやすいからである。
使うなら「何を伝えているか」を明示し、例外採用の理由を出力内に残すこと。

- **Glassmorphism**
  - 失敗しやすい理由: ぼかしが情報の境界とグループの帰属を曖昧にしやすい
  - 許容条件: 背景との差分、重なり順、固定レイヤーの関係を明確に示せるとき
  - 説明義務: どのレイヤー関係または状態境界を伝えているかを書く
- **アイコンとしての絵文字**
  - 失敗しやすい理由: 文脈と文化で意味が揺れ、操作対象の安定した記号になりにくい
  - 許容条件: 既存ブランドやコミュニティ文脈で意味が固定され、補助ラベルが付くとき
  - 説明義務: 絵文字単体に頼らず何の意味を補助しているかを書く
- **主テーマとしてのグラデーション**
  - 失敗しやすい理由: 雰囲気だけを増やし、情報階層やカテゴリを伝えないまま面積を支配しやすい
  - 許容条件: ブランド制約、密度表現、カテゴリ差、データの連続量を伝える役割があるとき
  - 説明義務: どの連続性や区別を伝えるために使うかを書く
- **装飾的な影**
  - 失敗しやすい理由: 階層や高さの意味を持たない奥行きが、重要度と物理感を誤読させやすい
  - 許容条件: 固定ヘッダー、ドラッグ対象、浮遊レイヤー、優先面の前後関係を示すとき
  - 説明義務: どの高さ関係または操作可能性を示しているかを書く
- **純粋に装飾的なホバーエフェクト**
  - 失敗しやすい理由: アフォーダンスや状態変化を伴わない動きが、意味のない刺激になる
  - 許容条件: クリック可能性、選択状態、因果関係、プレビュー対象を明確にする場合
  - 説明義務: ホバーで何の状態差分を伝えるかを書く
- **雰囲気としての色**
  - 失敗しやすい理由: カテゴリ、状態、優先度、関係性を持たない色が認知ルールを濁らせる
  - 許容条件: ブランド制約があり、同時に状態・カテゴリ・強調の規則としても一貫運用できるとき
  - 説明義務: その色が何を区別し、どの規則で再利用されるかを書く
- **構造的理由のない丸いカード**
  - 失敗しやすい理由: 柔らかさだけを足して関係や境界の意味を説明しない
  - 許容条件: タップ領域、グループ境界、状態コンテナとして角の意味が明確なとき
  - 説明義務: なぜその角丸が必要で、何の境界を伝えるかを書く
- **opacity-based text color**
  - 失敗しやすい理由: 背景との関係が不安定で、コントラスト設計を壊しやすい
  - 許容条件: 実背景込みで測定し、全状態で十分なコントラストが担保できるとき
  - 説明義務: どの背景で検証し、どの帯域に置くかを書く
- **flat contrast scale**
  - 失敗しやすい理由: どのテキストも似た濃さになり、優先度が潰れる
  - 許容条件: 意図的にフラットな読ませ方をする限定的な場面で、開始点が別手段で明確なとき
  - 説明義務: どの要素が主従を担保するかを書く
- **contrast as decoration**
  - 失敗しやすい理由: 「柔らかく見せたい」だけの低コントラストは意味を伝えない
  - 許容条件: 情報優先度や状態の差を、他の明確な信号と組み合わせて説明できるとき
  - 説明義務: 低コントラスト化で何を伝え、何で補完するかを書く
- **uniform spacing throughout**
  - 失敗しやすい理由: どこも同じ余白だと、どこがまとまりか分からない
  - 許容条件: 同じ意味レベルの反復構造だけで構成される場面
  - 説明義務: どのまとまりが等価で、どこで階層を立てるかを書く
- **arbitrary spacing values**
  - 失敗しやすい理由: その場しのぎの数値は偶発的な階層を生む
  - 許容条件: 既存 scale では説明できない例外があり、再利用基準を明文化するとき
  - 説明義務: なぜ既存 scale では足りず、その値が何を示すかを書く
- **near-identical spacing variants**
  - 失敗しやすい理由: 意味の差がない微差はノイズになる
  - 許容条件: 反復密度を段階制御する明確なルールがあるとき
  - 説明義務: どの差分がどの意味段階に対応するかを書く
- **spacing as decoration**
  - 失敗しやすい理由: 「きれいに見えるから」だけで gap を足すと関係の読みが崩れる
  - 許容条件: 余白がグルーピング、焦点化、密度制御のいずれかを担うとき
  - 説明義務: その gap が何の関係を伝えるかを書く
- **empty space without function**
  - 失敗しやすい理由: 分離・焦点化・密度調整を説明できない空白は意味を持たない
  - 許容条件: 視線の休止、重要ブロックの隔離、スクロールの節目として機能するとき
  - 説明義務: その空白がどの読み順または認知負荷制御に効くかを書く
- **collapsed hierarchy**
  - 失敗しやすい理由: 異なる意味レベルに同じ余白や同じコントラストを使い、構造を潰す
  - 許容条件: フラットであること自体が機能要件で、別の信号で優先順位を保証できるとき
  - 説明義務: どの別信号で階層を補うかを書く

## Sub-skills

`ma` は設計・実装のスキルである。審査・改善は専用サブスキルに委ねる。

| スキル | 用途 |
|---|---|
| `ma-review` | 完成UIの総合監査 |
| `ma-system` | デザインシステムの一貫性 |
| `ma-legibility` | 可読性・アクセシビリティ |
| `ma-mapping` | コントロールとターゲットの関係 |
| `ma-flow` | 視線の流れと情報順序 |
| `ma-reduction` | 不要な複雑さの除去 |

## Output Contract

このスキルの出力は、常に次の骨子で返すこと。

### 必須セクション

- 画面またはコンポーネントの目的
- 情報構造
- 主要状態と主要な状態遷移
- CTA とその優先順位
- 実装上の構成案
- デザイン根拠

### デザイン根拠に含めること

- 何を先に見せるか
- 何を後に見せるか
- なぜその視覚ルールが必要か
- 何を削るべきか
- 例外的に高リスク表現を採用した場合、それが何を伝え、なぜ必要か
- Explain the spacing logic: what the spacing scale is, where rhythm is kept, and where it is intentionally broken.
- contrast rationale
  - どのコントラスト帯を使うか
  - どの背景に対して検証するか
  - 非標準背景があるなら、どう確認するか
- spacing rationale
  - spacing scale
  - grouping logic
  - rhythm logic
  - どこで negative space を使って密度や焦点を制御するか
- 必要なら次にどのサブスキルでレビューすべきか

どれか一部だけを返してはならない。
哲学だけで終わらず、必ず実装指向の構成案まで含めること。

## Example

入力: 「分析ダッシュボードを設計して。数値だけでなく、何を見るべきかが分かるようにしたい。余白とコントラストの根拠もほしい」

期待: エンティティ、状態、優先順位、主要CTAを整理したうえで、実装方針とデザイン根拠まで返す。
その根拠には、どの情報をどのコントラスト帯に置くか、どこで余白を使ってグルーピングするかも含める。

追加例は `references/examples.md` を参照すること。
